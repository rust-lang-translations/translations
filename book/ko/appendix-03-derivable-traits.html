The Rust Programming Language

부록 C: 파생 가능한 트레이트

이 책의 여러 곳에서 우리는 구조체나 열거형 정의에 적용할 수 있는 derive 속성에 대해 논의했습니다. derive 속성은 여러분이 derive 구문으로 어노테이션을 단 타입에 대해 트레이트의 기본 구현을 생성하는 코드를 만들어 줍니다.

이 부록에서는 derive와 함께 사용할 수 있는 표준 라이브러리의 모든 트레이트에 대한 참조를 제공합니다. 각 섹션에서는 다음 내용을 다룹니다.

  • 이 트레이트를 파생(deriving)함으로써 사용할 수 있게 되는 연산자와 메서드
  • derive에 의해 제공되는 트레이트 구현이 하는 일
  • 이 트레이트를 구현하는 것이 해당 타입에 대해 무엇을 의미하는지
  • 트레이트 구현이 허용되거나 허용되지 않는 조건
  • 해당 트레이트를 필요로 하는 연산의 예시

derive 속성이 제공하는 것과 다른 동작을 원한다면, 각 트레이트를 수동으로 구현하는 방법에 대한 자세한 내용은 표준 라이브러리 문서를 참조하세요.

여기에 나열된 트레이트들은 표준 라이브러리에서 정의한 것들 중 derive를 사용하여 여러분의 타입에 구현할 수 있는 유일한 것들입니다. 표준 라이브러리에 정의된 다른 트레이트들은 합리적인 기본 동작을 가지고 있지 않으므로, 여러분이 달성하고자 하는 목적에 맞게 직접 구현해야 합니다.

derive 할 수 없는 트레이트의 예로, 최종 사용자를 위한 포맷팅을 처리하는 Display가 있습니다. 타입을 사용자에게 표시할 때는 항상 적절한 방식을 고려해야 합니다. 타입의 어떤 부분을 사용자에게 보여주어야 할까요? 사용자가 어떤 부분에 관심을 가질까요? 어떤 데이터 포맷이 사용자에게 가장 적절할까요? 러스트 컴파일러는 이러한 통찰력을 가지고 있지 않으므로, 여러분을 대신해 적절한 기본 동작을 제공할 수 없습니다.

이 부록에 나열된 파생 가능한 트레이트 목록이 전부는 아닙니다. 라이브러리들은 자신의 트레이트에 대해 derive를 구현할 수 있으므로, derive와 함께 사용할 수 있는 트레이트 목록은 사실상 무한합니다. derive를 구현하는 과정에는 20장의 “매크로” 섹션에서 다루는 절차적 매크로를 사용하는 것이 포함됩니다.

프로그래머 출력을 위한 Debug

Debug 트레이트는 포맷 문자열에서 디버그 포맷팅을 가능하게 하며, {} 자리표시자 안에 :?를 추가하여 이를 지정합니다.

Debug 트레이트를 사용하면 디버깅 목적으로 타입의 인스턴스를 출력할 수 있으므로, 여러분과 여러분의 타입을 사용하는 다른 프로그래머들이 프로그램 실행 중 특정 시점에 인스턴스의 상태를 검사할 수 있습니다.

Debug 트레이트는 예를 들어 assert_eq! 매크로를 사용하는 데 필요합니다. 이 매크로는 상등 단언(equality assertion)이 실패할 경우 인수로 주어진 인스턴스들의 값을 출력하여, 프로그래머가 왜 두 인스턴스가 같지 않은지 확인할 수 있도록 합니다.

상등 비교를 위한 PartialEq와 Eq

PartialEq 트레이트를 사용하면 타입의 인스턴스들을 비교하여 같은지 확인할 수 있으며, == 및 != 연산자를 사용할 수 있게 됩니다.

PartialEq를 파생하면 eq 메서드가 구현됩니다. 구조체에서 PartialEq를 파생할 경우, 모든 필드가 같아야만 두 인스턴스가 같은 것으로 간주되며, 어느 한 필드라도 다르면 같지 않은 것으로 간주됩니다. 열거형에서 파생할 경우, 각 변형(variant)은 자기 자신과만 같고 다른 변형과는 같지 않습니다.

PartialEq 트레이트는 예를 들어 assert_eq! 매크로를 사용하는 데 필요한데, 이 매크로는 타입의 두 인스턴스가 같은지 비교할 수 있어야 하기 때문입니다.

Eq 트레이트에는 메서드가 없습니다. 이 트레이트의 목적은 어노테이션이 달린 타입의 모든 값에 대해, 그 값이 자기 자신과 같음을 나타내는 것입니다. Eq 트레이트는 PartialEq를 구현하는 타입에만 적용할 수 있지만, PartialEq를 구현한다고 해서 모든 타입이 Eq를 구현할 수 있는 것은 아닙니다. 한 가지 예로 부동 소수점 숫자 타입이 있습니다. 부동 소수점 숫자의 구현 명세에 따르면, 숫자가 아님(NaN)을 나타내는 두 값은 서로 같지 않습니다.

Eq가 필요한 예로 HashMap<K, V>의 키가 있습니다. 그래야 HashMap<K, V>가 두 키가 같은지 판별할 수 있기 때문입니다.

순서 비교를 위한 PartialOrd와 Ord

PartialOrd 트레이트를 사용하면 정렬 목적으로 타입의 인스턴스들을 비교할 수 있습니다. PartialOrd를 구현하는 타입은 <, >, <=, >= 연산자와 함께 사용될 수 있습니다. PartialOrd 트레이트는 PartialEq를 구현하는 타입에만 적용할 수 있습니다.

PartialOrd를 파생하면 partial_cmp 메서드가 구현되는데, 이 메서드는 주어진 값들 사이에 순서가 정의되지 않을 경우 None이 되는 Option<Ordering>을 반환합니다. 해당 타입의 대부분의 값들은 비교 가능하지만 순서가 정의되지 않는 값의 예로는 부동 소수점의 NaN 값이 있습니다. 어떤 부동 소수점 숫자와 NaN 값을 인수로 하여 partial_cmp를 호출하면 None이 반환됩니다.

구조체에서 파생된 경우, PartialOrd는 구조체 정의에 필드가 나타나는 순서대로 각 필드의 값을 비교하여 두 인스턴스를 비교합니다. 열거형에서 파생된 경우, 열거형 정의에서 앞쪽에 선언된 변형(variant)이 뒤쪽에 나열된 변형보다 작은 것으로 간주됩니다.

PartialOrd 트레이트는 예를 들어, 범위 표현식으로 지정된 범위 내에서 임의의 값을 생성하는 rand 크레이트의 gen_range 메서드에서 필요합니다.

Ord 트레이트를 사용하면 어노테이션이 달린 타입의 어떤 두 값에 대해서도 유효한 순서가 존재함을 알 수 있습니다. Ord 트레이트는 cmp 메서드를 구현하는데, 유효한 순서가 항상 가능하므로 Option<Ordering>이 아닌 Ordering을 반환합니다. Ord 트레이트는 PartialOrd와 Eq를 모두 구현하는 타입에만 적용할 수 있습니다 (그리고 Eq는 PartialEq를 필요로 합니다). 구조체와 열거형에서 파생된 경우, cmp는 PartialOrd에서 파생된 partial_cmp 구현과 동일하게 동작합니다.

Ord가 필요한 예로는 값의 정렬 순서에 따라 데이터를 저장하는 데이터 구조인 BTreeSet<T>에 값을 저장하는 경우가 있습니다.

값 복제를 위한 Clone과 Copy

Clone 트레이트를 사용하면 값의 깊은 복사(deep copy)를 명시적으로 생성할 수 있으며, 복제 과정에서 임의의 코드가 실행되거나 힙 데이터가 복사될 수 있습니다. Clone에 대한 자세한 내용은 4장의 “변수와 데이터가 상호작용하는 방식: 클론”을 참조하세요.

Clone을 파생하면 clone 메서드가 구현되는데, 타입 전체에 대해 구현될 때 타입의 각 구성 요소에 대해 clone을 호출합니다. 즉, Clone을 파생하려면 타입의 모든 필드나 값들도 Clone을 구현해야 함을 의미합니다.

Clone이 필요한 예로는 슬라이스에서 to_vec 메서드를 호출하는 경우가 있습니다. 슬라이스는 자신이 포함하고 있는 타입 인스턴스들을 소유하지 않지만, to_vec에서 반환된 벡터는 인스턴스들을 소유해야 하므로 각 아이템에 대해 clone을 호출합니다. 따라서 슬라이스에 저장된 타입은 반드시 Clone을 구현해야 합니다.

Copy 트레이트를 사용하면 스택에 저장된 비트만 복사하여 값을 복제할 수 있으며, 어떠한 임의의 코드도 필요하지 않습니다. Copy에 대한 자세한 내용은 4장의 “스택 전용 데이터: 복사”를 참조하세요.

Copy 트레이트는 프로그래머가 메서드를 오버로딩하여 임의의 코드가 실행되지 않는다는 가정을 위반하는 것을 방지하기 위해 어떠한 메서드도 정의하지 않습니다. 이런 방식으로 모든 프로그래머는 값을 복사하는 것이 매우 빠를 것이라고 가정할 수 있습니다.

모든 구성 요소가 Copy를 구현하는 타입이라면 무엇이든 Copy를 파생할 수 있습니다. Copy를 구현하는 타입은 반드시 Clone도 구현해야 하는데, Copy를 구현하는 타입은 Copy와 동일한 작업을 수행하는 단순한 Clone 구현을 가지기 때문입니다.

Copy 트레이트가 필수인 경우는 드뭅니다. Copy를 구현하는 타입은 최적화가 가능하므로 clone을 호출할 필요가 없으며, 이는 코드를 더 간결하게 만듭니다.

Copy로 가능한 모든 작업은 Clone으로도 수행할 수 있지만, 코드가 더 느려지거나 곳곳에서 clone을 사용해야 할 수 있습니다.

값을 고정 크기 값으로 매핑하기 위한 Hash

Hash 트레이트를 사용하면 임의 크기의 타입 인스턴스를 해시 함수를 사용하여 고정 크기의 값으로 매핑할 수 있습니다. Hash를 파생하면 hash 메서드가 구현됩니다. 파생된 hash 메서드 구현은 타입의 각 구성 요소에 대해 hash를 호출한 결과를 결합합니다. 즉, Hash를 파생하려면 모든 필드나 값들도 Hash를 구현해야 함을 의미합니다.

Hash가 필요한 예로는 데이터를 효율적으로 저장하기 위해 HashMap<K, V>의 키로 사용하는 경우가 있습니다.

기본값을 위한 Default

Default 트레이트를 사용하면 타입에 대한 기본값을 생성할 수 있습니다. Default를 파생하면 default 함수가 구현됩니다. 파생된 default 함수 구현은 타입의 각 구성 요소에 대해 default 함수를 호출합니다. 즉, Default를 파생하려면 타입의 모든 필드나 값들도 Default를 구현해야 함을 의미합니다.

Default::default 함수는 5장의 “구조체 업데이트 구문을 사용하여 기존 인스턴스로부터 새로운 인스턴스 생성하기”에서 다룬 구조체 업데이트 구문과 함께 자주 사용됩니다. 구조체의 몇몇 필드만 커스터마이징하고 나머지 필드들은 ..Default::default()를 사용하여 기본값으로 설정하고 사용할 수 있습니다.

Default 트레이트는 예를 들어 Option<T> 인스턴스에서 unwrap_or_default 메서드를 사용할 때 필요합니다. 만약 Option<T>가 None인 경우, unwrap_or_default 메서드는 Option<T>에 저장된 타입 T에 대한 Default::default 결과를 반환합니다.
