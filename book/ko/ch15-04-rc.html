The Rust Programming Language

Rc<T>, 참조 카운팅 스마트 포인터

대부분의 경우 소유권은 명확합니다: 어떤 변수가 주어진 값을 소유하는지 정확히 알 수 있습니다. 하지만 하나의 값이 여러 소유자를 가질 수 있는 경우도 있습니다. 예를 들어, 그래프 데이터 구조에서 여러 에지(edge)가 동일한 노드를 가리킬 수 있으며, 그 노드는 개념적으로 자신을 가리키는 모든 에지에 의해 소유됩니다. 노드는 자신을 가리키는 에지가 하나도 없어서 소유자가 없을 때만 정리되어야 합니다.

러스트의 Rc<T> 타입을 사용하여 명시적으로 다중 소유권을 활성화해야 합니다. Rc<T>는 _참조 카운팅(reference counting)_의 약자입니다. Rc<T> 타입은 값이 여전히 사용 중인지 결정하기 위해 그 값에 대한 참조 횟수를 추적합니다. 만약 값에 대한 참조가 0개라면, 어떤 참조도 무효화되지 않은 채로 그 값을 정리할 수 있습니다.

Rc<T>를 거실에 있는 TV라고 상상해 보세요. 한 사람이 TV를 보러 들어오면 TV를 켭니다. 다른 사람들도 방에 들어와 TV를 볼 수 있습니다. 마지막 사람이 방을 나갈 때, 더 이상 사용되지 않으므로 TV를 끕니다. 만약 다른 사람들이 여전히 TV를 보고 있는데 누군가 TV를 꺼버린다면, 남아있는 시청자들이 가만히 있지 않겠죠!

프로그램의 여러 부분에서 읽을 수 있도록 힙에 데이터를 할당하고 싶지만, 컴파일 타임에 어떤 부분이 마지막으로 그 데이터를 사용할지 결정할 수 없을 때 Rc<T> 타입을 사용합니다. 만약 어떤 부분이 마지막에 끝날지 안다면, 그 부분을 데이터의 소유자로 만들면 컴파일 타임에 강제되는 일반적인 소유권 규칙이 적용될 것입니다.

Rc<T>는 오직 단일 스레드 시나리오에서만 사용 가능하다는 점에 유의하세요. 16장에서 동시성을 논의할 때, 멀티스레드 프로그램에서 참조 카운팅을 하는 방법을 다룰 것입니다.

Rc<T>를 사용하여 데이터 공유하기

목록 15-5의 콘스 리스트 예제로 돌아가 봅시다. 우리가 이를 Box<T>를 사용하여 정의했음을 상기해 보세요. 이번에는 세 번째 리스트의 소유권을 공유하는 두 개의 리스트를 만들 것입니다. 개념적으로는 그림 15-3과 유사한 모습입니다:

그림 15-3: 세 번째 리스트 a를 공유 소유하는 두 리스트 b와 c

5와 10을 포함하는 리스트 a를 생성할 것입니다. 그런 다음 3으로 시작하는 리스트 b와 4로 시작하는 리스트 c를 만들 것입니다. 리스트 b와 c 모두 5와 10을 포함하는 첫 번째 리스트 a로 이어지게 됩니다. 즉, 두 리스트 모두 5와 10을 포함하는 첫 번째 리스트를 공유하게 됩니다.

Box<T>를 사용한 우리의 List 정의로 이 시나리오를 구현하려는 시도는 목록 15-17에서 보여주듯이 작동하지 않을 것입니다:

enum List {
    Cons(i32, Box<List>),
    Nil,
}

use crate::List::{Cons, Nil};

fn main() {
    let a = Cons(5, Box::new(Cons(10, Box::new(Nil))));
    let b = Cons(3, Box::new(a));
    let c = Cons(4, Box::new(a));
}

Cons 변종은 자신이 보유한 데이터를 소유하므로, 리스트 b를 생성할 때 a는 b로 이동하고 b가 a를 소유하게 됩니다. 그런 다음 c를 생성할 때 a를 다시 사용하려고 시도하면, a가 이미 이동되었기 때문에 허용되지 않습니다.

Cons가 참조를 보유하도록 정의를 변경할 수도 있지만, 그렇게 하면 라이프타임 매개변수를 지정해야 합니다. 라이프타임 매개변수를 지정한다는 것은 리스트의 모든 요소가 전체 리스트만큼은 오래 살 것임을 명시하는 것입니다. 목록 15-17의 요소들과 리스트들이 바로 그런 경우지만, 모든 시나리오가 그렇지는 않습니다.

대신, 목록 15-18에서처럼 Box<T> 대신 Rc<T>를 사용하도록 List 정의를 변경하겠습니다. 이제 각 Cons 변종은 값 하나와 List를 가리키는 Rc<T>를 보유하게 됩니다. b를 생성할 때 a의 소유권을 가져오는 대신, a가 보유하고 있는 Rc<List>를 복제(clone)할 것입니다. 이렇게 하면 참조 횟수가 1에서 2로 증가하며, a와 b가 그 Rc<List> 안의 데이터에 대한 소유권을 공유하게 됩니다. 또한 c를 생성할 때도 a를 복제하여 참조 횟수를 2에서 3으로 늘릴 것입니다. Rc::clone을 호출할 때마다 Rc<List> 내부 데이터에 대한 참조 횟수가 증가하며, 참조 횟수가 0이 되지 않는 한 데이터는 정리되지 않을 것입니다.

enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    let b = Cons(3, Rc::clone(&a));
    let c = Cons(4, Rc::clone(&a));
}

Rc<T>는 프렐루드에 포함되어 있지 않으므로, 이를 스코프로 가져오기 위해 use 구문을 추가해야 합니다. main에서는 5와 10을 보유한 리스트를 생성하여 a에 새로운 Rc<List>로 저장합니다. 그런 다음 b와 c를 생성할 때, Rc::clone 함수를 호출하고 a에 있는 Rc<List>에 대한 참조를 인자로 전달합니다.

Rc::clone(&a) 대신 a.clone()을 호출할 수도 있었겠지만, 이 경우에는 Rc::clone을 사용하는 것이 러스트의 관례입니다. Rc::clone의 구현은 대부분의 타입들이 구현하는 clone처럼 모든 데이터를 깊은 복사(deep copy)하지 않습니다. Rc::clone 호출은 참조 횟수만 증가시키며, 이는 시간이 오래 걸리지 않습니다. 데이터의 깊은 복사는 많은 시간이 걸릴 수 있습니다. 참조 카운팅을 위해 Rc::clone을 사용함으로써, 우리는 깊은 복사류의 클론과 참조 횟수만 증가시키는 종류의 클론을 시각적으로 구분할 수 있습니다. 코드에서 성능 문제를 찾을 때, 깊은 복사 클론만 고려하면 되고 Rc::clone 호출은 무시할 수 있습니다.

Rc<T>를 복제하면 참조 횟수가 증가합니다

목록 15-18의 예제를 수정하여, a에 있는 Rc<List>에 대한 참조를 만들고 드롭함에 따라 참조 횟수가 어떻게 변하는지 확인해 봅시다.

목록 15-19에서는 main을 수정하여 리스트 c를 감싸는 내부 스코프를 만들겠습니다. 그러면 c가 스코프를 벗어날 때 참조 횟수가 어떻게 변하는지 볼 수 있습니다.

enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    println!("a 생성 후의 횟수 = {}", Rc::strong_count(&a));
    let b = Cons(3, Rc::clone(&a));
    println!("b 생성 후의 횟수 = {}", Rc::strong_count(&a));
    {
        let c = Cons(4, Rc::clone(&a));
        println!("c 생성 후의 횟수 = {}", Rc::strong_count(&a));
    }
    println!("c가 스코프를 벗어난 후의 횟수 = {}", Rc::strong_count(&a));
}

At each point in the program where the reference count changes, we print the reference count, which we get by calling the Rc::strong_count function. This function is named strong_count rather than count because the Rc<T> type also has a weak_count; we’ll see what weak_count is used for in “Preventing Reference Cycles: Turning an Rc<T> into a Weak<T>”.

이 코드는 다음과 같이 출력합니다:

$ cargo run
   Compiling cons-list v0.1.0 (file:///projects/cons-list)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.45s
     Running `target/debug/cons-list`
count after creating a = 1
count after creating b = 2
count after creating c = 3
count after c goes out of scope = 2

a에 있는 Rc<List>의 초기 참조 횟수가 1임을 알 수 있습니다. 그 후 clone을 호출할 때마다 횟수가 1씩 증가합니다. c가 스코프를 벗어나면 횟수가 1 감소합니다. 참조 횟수를 늘리기 위해 Rc::clone을 호출해야 하는 것과 달리, 참조 횟수를 줄이기 위해 어떤 함수를 호출할 필요는 없습니다: Rc<T> 값이 스코프를 벗어날 때 Drop 트레이트의 구현이 자동으로 참조 횟수를 감소시킵니다.

이 예제에서 볼 수 없는 점은 main의 끝에서 b와 a가 스코프를 벗어날 때 참조 횟수가 0이 되고, Rc<List>가 완전히 정리된다는 것입니다. Rc<T>를 사용하면 하나의 값이 여러 명의 소유자를 가질 수 있으며, 참조 횟수는 소유자 중 누구라도 남아있는 한 값이 유효하게 유지됨을 보장합니다.

불변 참조를 통해, Rc<T>는 프로그램의 여러 부분 사이에서 오직 읽기 전용으로만 데이터를 공유할 수 있게 해줍니다. 만약 Rc<T>가 여러 개의 가변 참조도 허용했다면, 4장에서 논의했던 대여 규칙 중 하나를 위반하게 될 것입니다: 동일한 위치에 대한 여러 개의 가변 대여는 데이터 경합(data races)과 부정합을 일으킬 수 있습니다. 하지만 데이터를 수정할 수 있는 것은 매우 유용합니다! 다음 섹션에서는 이러한 불변성 제한과 함께 작업하기 위해 Rc<T>와 결합하여 사용할 수 있는 내부 가변성 패턴과 RefCell<T> 타입에 대해 논의하겠습니다.
