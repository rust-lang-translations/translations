crates.io에 게시하기

세상과 공유하고 싶은 라이브러리가 생겼다면 이제 crates.io에 게시할 시간입니다! 크레이트를 게시한다는 것은 특정 버전을 crates.io에 호스팅하기 위해 업로드하는 것입니다.

크레이트를 게시할 때는 주의하세요. 게시는 영구적이기 때문입니다. 버전은 덮어쓸 수 없으며 코드는 삭제할 수 없습니다. 그러나 게시할 수 있는 버전 수에는 제한이 없습니다.

첫 게시 전

먼저 API 토큰을 얻으려면 crates.io에 계정이 필요합니다. 그렇게 하려면 홈페이지를 방문하여 GitHub 계정을 통해 로그인하세요(현재 필수). 또한 계정 설정 페이지에서 이메일 주소를 제공하고 확인해야 합니다. 완료되면 API 토큰을 생성하고 복사해야 합니다. 페이지를 떠나면 다시 볼 수 없습니다.

그런 다음 cargo login 명령어를 실행하세요.

$ cargo login

그런 다음 프롬프트에 지정된 토큰을 입력하세요.

please paste the API Token found on https://crates.io/me below
abcdefghijklmnopqrstuvwxyz012345

이 명령어는 Cargo에 API 토큰을 알리고 로컬의 ~/.cargo/credentials.toml에 저장합니다. 이 토큰은 비밀이며 다른 사람과 공유해서는 안 됩니다. 어떤 이유로든 유출되면 즉시 해지해야 합니다.

참고: cargo logout 명령어를 사용하여 credentials.toml에서 토큰을 제거할 수 있습니다. 로컬 컴퓨터에 더 이상 저장할 필요가 없는 경우 유용할 수 있습니다.

새 크레이트 게시 전

crates.io의 크레이트 이름은 선착순으로 할당된다는 점을 명심하세요. 크레이트 이름이 사용되면 다른 크레이트에 사용할 수 없습니다.

Cargo.toml에서 지정할 수 있는 메타데이터를 확인하여 크레이트를 더 쉽게 찾을 수 있도록 하세요! 게시하기 전에 다음 필드를 채웠는지 확인하세요:

  • license 또는 license-file
  • 설명
  • 홈페이지
  • 저장소
  • README

필수는 아니지만 keywords 및 categories를 포함하는 것도 좋은 생각입니다.

라이브러리를 게시하는 경우 Rust API 가이드라인을 참조할 수도 있습니다.

크레이트 패키징하기

다음 단계는 크레이트를 패키징하여 crates.io에 업로드하는 것입니다. 이를 위해 cargo publish 하위 명령을 사용합니다. 이 명령어는 다음 단계를 수행합니다:

  1. 패키지에 대한 일부 검증 검사를 수행합니다.
  2. 소스 코드를 .crate 파일로 압축합니다.
  3. .crate` 파일을 임시 디렉토리에 추출하고 컴파일되는지 확인합니다.
  4. .crate 파일을 crates.io에 업로드합니다.
  5. 레지스트리는 업로드된 패키지를 추가하기 전에 몇 가지 추가 검사를 수행합니다.

게시하기 전에 경고나 오류가 없는지 확인하기 위해 먼저 cargo publish --dry-run(또는 동등한 cargo package)을 실행하는 것이 좋습니다. 이렇게 하면 위에 나열된 처음 세 단계가 수행됩니다.

$ cargo publish --dry-run

생성된 .crate 파일은 target/package 디렉토리에서 검사할 수 있습니다. crates.io는 현재 .crate 파일에 10MB 크기 제한이 있습니다. .crate 파일의 크기를 확인하여 테스트 데이터, 웹사이트 문서 또는 코드 생성과 같이 패키지를 빌드하는 데 필요하지 않은 큰 자산을 실수로 패키징하지 않았는지 확인할 수 있습니다. 다음 명령으로 포함된 파일을 확인할 수 있습니다.

$ cargo package --list

Cargo는 패키징할 때 버전 관리 시스템에서 무시된 파일을 자동으로 무시하지만, 무시할 추가 파일 집합을 지정하려면 매니페스트에서 exclude 키를 사용할 수 있습니다.

[package]
# ...
exclude = [
    "public/assets/*",
    "videos/*",
]

If you’d rather explicitly list the files to include, Cargo also supports an include key, which if set, overrides the exclude key:

[package]
# ...
include = [
    "**/*.rs",
]

크레이트 업로드하기

게시할 준비가 되면 cargo publish 명령을 사용하여 crates.io에 업로드합니다.

$ cargo publish

이것으로 첫 번째 크레이트를 게시했습니다!

기존 크레이트의 새 버전 게시하기

새 버전을 릴리스하려면 Cargo.toml 매니페스트에 지정된 version 값을 변경하세요. 호환되는 변경 사항이 무엇인지에 대한 지침을 제공하는 SemVer 규칙을 염두에 두세요. 그런 다음 위에서 설명한 대로 cargo publish를 실행하여 새 버전을 업로드하세요.

권장 사항: 전체 릴리스 프로세스를 고려하고 가능한 한 자동화하세요.

각 버전에는 다음이 포함되어야 합니다:

  • 변경 로그 항목, 수동으로 관리하는 것이 좋지만 생성된 것이 없는 것보다는 낫습니다.
  • 게시된 커밋을 가리키는 git 태그

다양한 워크플로를 대표하는 타사 도구의 예는 다음과 같습니다(알파벳순):

  • cargo-release
  • cargo-smart-release
  • release-plz

자세한 내용은 crates.io를 참조하세요.

crates.io 기반 크레이트 관리하기

크레이트 관리는 주로 crates.io 웹 인터페이스보다는 명령줄 cargo 도구를 통해 이루어집니다. 이를 위해 크레이트를 관리하는 몇 가지 하위 명령이 있습니다.

cargo yank

구문 오류, 파일 포함 누락 등 여러 가지 이유로 인해 크레이트 버전이 손상되는 경우가 발생할 수 있습니다. 이러한 상황을 위해 Cargo는 크레이트 버전의 “yank”를 지원합니다.

$ cargo yank --version 1.0.1
$ cargo yank --version 1.0.1 --undo

yank는 코드를 삭제하지 않습니다. 이 기능은 실수로 업로드된 비밀을 삭제하기 위한 것이 아닙니다. 예를 들어, 그런 일이 발생하면 즉시 해당 비밀을 재설정해야 합니다.

yank된 버전의 의미는 해당 버전에 대해 새로운 의존성을 생성할 수 없지만, 기존의 모든 의존성은 계속 작동한다는 것입니다. crates.io의 주요 목표 중 하나는 시간이 지나도 변경되지 않는 크레이트의 영구 아카이브 역할을 하는 것이며, 버전 삭제를 허용하는 것은 이 목표에 위배됩니다. 본질적으로 yank는 Cargo.lock이 있는 모든 패키지는 손상되지 않지만, 향후 생성될 Cargo.lock 파일에는 yank된 버전이 나열되지 않음을 의미합니다.

cargo owner

크레이트는 종종 한 명 이상의 개발자가 개발하거나, 주 유지보수자가 시간이 지남에 따라 변경될 수 있습니다! 크레이트 소유자는 크레이트의 새 버전을 게시할 수 있는 유일한 사람이지만, 소유자는 추가 소유자를 지정할 수 있습니다.

$ cargo owner --add github-handle
$ cargo owner --remove github-handle
$ cargo owner --add github:rust-lang:owners
$ cargo owner --remove github:rust-lang:owners

이 명령에 제공되는 소유자 ID는 GitHub 사용자 이름 또는 GitHub 팀이어야 합니다.

--add에 사용자 이름이 주어지면 해당 사용자는 크레이트에 대한 모든 권한을 가진 “지정된” 소유자로 초대됩니다. 크레이트 버전을 게시하거나 yank할 수 있을 뿐만 아니라, 소유자를 추가하거나 제거할 수 있으며, 자신을 소유자로 만든 소유자도 포함됩니다. 말할 필요도 없이, 완전히 신뢰하지 않는 사람을 지정된 소유자로 만들어서는 안 됩니다. 지정된 소유자가 되려면 사용자는 이전에 crates.io에 로그인해야 합니다.

--add에 팀 이름이 주어지면 해당 팀은 크레이트에 대한 제한된 권한을 가진 “팀” 소유자로 초대됩니다. 이들은 크레이트 버전을 게시하거나 yank할 수 있는 권한은 있지만, 소유자를 추가하거나 제거할 수 있는 권한은 없습니다. 소유자 그룹을 관리하는 데 더 편리할 뿐만 아니라, 팀은 소유자가 악의적으로 변하는 것에 대해 약간 더 안전합니다.

팀에 대한 구문은 현재 github:org:team입니다(위 예시 참조). 팀을 소유자로 초대하려면 해당 팀의 구성원이어야 합니다. 팀을 소유자에서 제거하는 데는 이러한 제한이 적용되지 않습니다.

GitHub 권한

팀 멤버십은 GitHub에서 간단한 공개 액세스를 제공하는 것이 아니므로, 팀과 작업할 때 다음 메시지를 만날 가능성이 높습니다:

이 요청을 완료하는 데 필요한 속성을 GitHub에서 쿼리할 권한이 없는 것 같습니다. GitHub 조직 멤버십을 읽을 권한을 부여하려면 crates.io에서 다시 인증해야 할 수 있습니다.

이것은 기본적으로 “팀을 쿼리하려고 했지만, 멤버십 액세스 제어의 다섯 가지 수준 중 하나가 이를 거부했습니다”에 대한 포괄적인 설명입니다. 과장이 아닙니다. GitHub의 팀 액세스 제어 지원은 엔터프라이즈 등급입니다.

가장 가능성이 높은 원인은 이 기능이 추가되기 전에 마지막으로 로그인했기 때문입니다. 우리는 원래 사용자 인증 시 GitHub에 어떤 권한도 요청하지 않았습니다. 왜냐하면 사용자 토큰을 로그인 외에는 사용하지 않았기 때문입니다. 그러나 이제 사용자를 대신하여 팀 멤버십을 쿼리하려면 read:org 범위가 필요합니다.

이 범위에 대한 접근을 거부할 수 있으며, 팀이 도입되기 전에 작동했던 모든 기능은 계속 작동합니다. 그러나 팀을 소유자로 추가하거나 팀 소유자로 크레이트를 게시할 수는 없습니다. 이를 시도하면 위와 같은 오류가 발생합니다. 또한 소유하지 않은 크레이트를 게시하려고 시도했지만 팀이 있는 경우에도 이 오류가 발생할 수 있습니다.

마음을 바꾸거나 crates.io에 충분한 권한이 있는지 확실하지 않은 경우, 언제든지 https://crates.io/로 이동하여 다시 인증할 수 있습니다. 그러면 crates.io에 필요한 모든 범위가 없는 경우 권한을 요청하는 메시지가 표시됩니다.

GitHub를 쿼리하는 데 추가적인 장벽은 조직이 타사 액세스를 적극적으로 거부할 수 있다는 것입니다. 이를 확인하려면 다음으로 이동하세요:

https://github.com/organizations/:org/settings/oauth_application_policy

GitHub 팀 액세스 오류 문제 해결

GitHub 팀을 크레이트 소유자로 추가하려고 할 때 다음과 같은 오류가 발생할 수 있습니다:

error: failed to invite owners to crate <crate_name>: api errors (status 200 OK): could not find the github team org/repo

In that case, you should go to the GitHub Application settings page and check if crates.io is listed in the Authorized OAuth Apps tab. If it isn’t, you should go to https://crates.io/ and authorize it. Then go back to the Application Settings page on GitHub, click on the crates.io application in the list, and make sure you or your organization is listed in the “Organization access” list with a green check mark. If there’s a button labeled Grant or Request, you should grant the access or request the org owner to do so.
