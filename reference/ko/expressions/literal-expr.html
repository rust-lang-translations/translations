[expr.literal]

리터럴 표현식

[expr.literal.syntax]

구문
리터럴표현식 :
      문자리터럴
   | 문자열리터럴
   | 원시문자열리터럴
   | 바이트리터럴
   | 바이트문자열리터럴
   | 원시바이트문자열리터럴
   | C문자열리터럴
   | 원시C문자열리터럴
   | 정수리터럴
   | 부동소수점리터럴
   | true | false

[expr.literal.intro]

_리터럴 표현식(literal expression)_은 단일 토큰으로 구성된 표현식으로, 이름을 통해 참조하거나 다른 평가 규칙을 따르는 대신, 그것이 평가되는 값을 즉시 그리고 직접적으로 나타냅니다.

[expr.literal.const-expr]

리터럴은 상수 표현식의 한 형태이므로, (주로) 컴파일 타임에 평가됩니다.

[expr.literal.literal-token]

앞서 설명한 어휘적 리터럴 형식들은 각각 리터럴 표현식을 구성할 수 있으며, true와 false 키워드도 마찬가지입니다.

#![allow(unused)]
fn main() {
"hello";   // 문자열 타입
'5';       // 문자 타입
5;         // 정수 타입
}
[expr.literal.string-representation]

아래 설명에서 토큰의 _문자열 표현(string representation)_은 렉서 문법 스니펫의 토큰 생성물과 일치하는 입력 문자 시퀀스를 의미합니다.

참고: 이 문자열 표현에는 U+000D (CR) 뒤에 바로 U+000A (LF)가 오는 문자가 포함되지 않습니다. 이 쌍은 이전에 이미 단일 U+000A (LF)로 변환되었을 것이기 때문입니다.

[expr.literal.escape]

이스케이프

[expr.literal.escape.intro]

아래의 텍스트 리터럴 표현식에 대한 설명에서는 여러 형태의 _이스케이프(escape)_를 사용합니다.

[expr.literal.escape.sequence]

각 이스케이프 형태는 다음으로 특징지어집니다:

  • 이스케이프 시퀀스(escape sequence): 항상 U+005C (\)로 시작하는 문자 시퀀스
  • 이스케이프된 값(escaped value): 단일 문자 또는 빈 문자 시퀀스

아래의 이스케이프 정의에서:

  • _8진수 숫자(octal digit)_는 0-7 범위의 문자 중 하나입니다.
  • _16진수 숫자(hexadecimal digit)_는 0-9, a-f, 또는 A-F 범위의 문자 중 하나입니다.
[expr.literal.escape.simple]

단순 이스케이프

다음 표의 첫 번째 열에 나타나는 각 문자 시퀀스는 이스케이프 시퀀스입니다.

각 경우에, 이스케이프된 값은 두 번째 열의 해당 항목에 주어진 문자입니다.

이스케이프 시퀀스이스케이프된 값
\0U+0000 (NUL)
\tU+0009 (HT)
\nU+000A (LF)
\rU+000D (CR)
큰따옴표U+0022 (큰따옴표)
\'U+0027 (작은따옴표)
\\U+005C (역슬래시)
[expr.literal.escape.hex-octet]

8비트 이스케이프

이스케이프 시퀀스는 \x 뒤에 두 개의 16진수 숫자가 오는 것으로 구성됩니다.

이스케이프된 값은 이스케이프 시퀀스의 마지막 두 문자를 16진수 정수로 해석한 결과(기수가 16인 u8::from_str_radix를 사용한 것과 같음)를 유니코드 스칼라 값(Unicode scalar value)으로 가지는 문자입니다.

참고: 따라서 이스케이프된 값은 u8 범위 내의 유니코드 스칼라 값을 가집니다.

[expr.literal.escape.hex-ascii]

7비트 이스케이프

이스케이프 시퀀스는 \x 뒤에 8진수 숫자 하나와 16진수 숫자 하나가 차례로 오는 것으로 구성됩니다.

이스케이프된 값은 이스케이프 시퀀스의 마지막 두 문자를 16진수 정수로 해석한 결과(기수가 16인 u8::from_str_radix를 사용한 것과 같음)를 유니코드 스칼라 값(Unicode scalar value)으로 가지는 문자입니다.

[expr.literal.escape.unicode]

유니코드 이스케이프

이스케이프 시퀀스는 \u{ 뒤에 16진수 숫자 또는 _로 구성된 문자 시퀀스가 오고, 그 뒤에 }가 오는 것으로 구성됩니다.

이스케이프된 값은 이스케이프 시퀀스에 포함된 16진수 숫자들을 16진수 정수로 해석한 결과(기수가 16인 u32::from_str_radix를 사용한 것과 같음)를 유니코드 스칼라 값으로 가지는 문자입니다.

참고: 허용되는 문자리터럴(CHAR_LITERAL) 또는 문자열리터럴(STRING_LITERAL) 토큰의 형식은 그러한 문자가 반드시 존재함을 보장합니다.

[expr.literal.continuation]

문자열 연속 이스케이프

이스케이프 시퀀스는 \ 뒤에 즉시 U+000A (LF)가 오고, 그다음 비-공백 문자 전까지의 모든 공백 문자들로 구성됩니다. 이 목적을 위해 공백 문자는 U+0009 (HT), U+000A (LF), U+000D (CR), 그리고 U+0020 (SPACE)입니다.

이스케이프된 값은 빈 문자 시퀀스입니다.

참고: 이 형태의 이스케이프 효과는 문자열 연속 시퀀스가 뒤따르는 공백(추가적인 줄 바꿈 포함)을 건너뛴다는 것입니다. 따라서 a, b, c는 동일합니다.

#![allow(unused)]
fn main() {
let a = "foobar";
let b = "foo\
         bar";
let c = "foo\

     bar";

assert_eq!(a, b);
assert_eq!(b, c);
}

추가적인 줄 바꿈을 건너뛰는 것(예시 c와 같은 경우)은 잠재적으로 혼란스럽고 예상치 못한 일일 수 있습니다. 이 동작은 미래에 조정될 수 있습니다. 결정이 내려지기 전까지는 라인 연속 기능을 사용하여 여러 줄 바꿈을 건너뛰는 것에 의존하지 않는 것이 권장됩니다. 자세한 내용은 이 이슈를 참조하세요.

[expr.literal.char]

문자 리터럴 표현식

[expr.literal.char.intro]

문자 리터럴 표현식은 단일 문자리터럴(CHAR_LITERAL) 토큰으로 구성됩니다.

[expr.literal.char.type]

표현식의 타입은 기본 char 타입입니다.

[expr.literal.char.no-suffix]

토큰은 접미사(suffix)를 가져서는 안 됩니다.

[expr.literal.char.literal-content]

토큰의 _리터럴 내용(literal content)_은 토큰의 문자열 표현에서 첫 번째 U+0027 (') 뒤에 오고 마지막 U+0027 (') 앞에 오는 문자 시퀀스입니다.

[expr.literal.char.represented]

리터럴 표현식의 _표현된 문자(represented character)_는 다음과 같이 리터럴 내용으로부터 유도됩니다:

[expr.literal.char.escape]
  • 리터럴 내용이 다음 형태의 이스케이프 시퀀스 중 하나인 경우, 표현된 문자는 해당 이스케이프 시퀀스의 이스케이프된 값입니다.
    • 단순 이스케이프
    • 7비트 이스케이프
    • 유니코드 이스케이프
[expr.literal.char.single]
  • 그렇지 않은 경우, 표현된 문자는 리터럴 내용을 구성하는 단일 문자입니다.
[expr.literal.char.result]

표현식의 값은 표현된 문자의 유니코드 스칼라 값에 해당하는 char입니다.

참고: 허용되는 문자리터럴(CHAR_LITERAL) 토큰의 형식은 이러한 규칙들이 항상 단일 문자를 생성함을 보장합니다.

문자 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
'R';                               // R
'\'';                              // '
'\x52';                            // R
'\u{00E6}';                        // 라틴어 소문자 AE (U+00E6)
}
[expr.literal.string]

문자열 리터럴 표현식

[expr.literal.string.intro]

문자열 리터럴 표현식은 단일 문자열리터럴(STRING_LITERAL) 또는 원시문자열리터럴(RAW_STRING_LITERAL) 토큰으로 구성됩니다.

[expr.literal.string.type]

표현식의 타입은 기본 str 타입에 대한 공유 참조(static 라이프타임을 가짐)입니다. 즉, 타입은 &'static str입니다.

[expr.literal.string.no-suffix]

토큰은 접미사(suffix)를 가져서는 안 됩니다.

[expr.literal.string.literal-content]

토큰의 _리터럴 내용(literal content)_은 토큰의 문자열 표현에서 첫 번째 U+0022 (") 뒤에 오고 마지막 U+0022 (") 앞에 오는 문자 시퀀스입니다.

[expr.literal.string.represented]

리터럴 표현식의 _표현된 문자열(represented string)_은 다음과 같이 리터럴 내용으로부터 유도된 문자 시퀀스입니다:

[expr.literal.string.escape]

  • 토큰이 문자열리터럴(STRING_LITERAL)인 경우, 리터럴 내용에서 나타나는 다음 형태의 각 이스케이프 시퀀스는 해당 이스케이프 시퀀스의 이스케이프된 값으로 대체됩니다.

    • 단순 이스케이프
    • 7비트 이스케이프
    • 유니코드 이스케이프
    • 문자열 연속 이스케이프 이러한 대체는 왼쪽에서 오른쪽 순서로 발생합니다. 예를 들어, "\\x41" 토큰은 문자 \, x, 4, 1로 변환됩니다.
[expr.literal.string.raw]
  • 토큰이 원시문자열리터럴(RAW_STRING_LITERAL)인 경우, 표현된 문자열은 리터럴 내용과 동일합니다.
[expr.literal.string.result]

표현식의 값은 표현된 문자열의 UTF-8 인코딩을 포함하는 정적으로 할당된 str에 대한 참조입니다.

문자열 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
"foo"; r"foo";                     // foo
"\"foo\""; r#""foo""#;             // "foo"

"foo #\"# bar";
r##"foo #"# bar"##;                // foo #"# bar

"\x52"; "R"; r"R";                 // R
"\\x52"; r"\x52";                  // \x52
}
[expr.literal.byte-char]

바이트 리터럴 표현식

[expr.literal.byte-char.intro]

바이트 리터럴 표현식은 단일 바이트리터럴(BYTE_LITERAL) 토큰으로 구성됩니다.

[expr.literal.byte-char.literal]

표현식의 타입은 기본 u8 타입입니다.

[expr.literal.byte-char.no-suffix]

토큰은 접미사(suffix)를 가져서는 안 됩니다.

[expr.literal.byte-char.literal-content]

토큰의 _리터럴 내용(literal content)_은 토큰의 문자열 표현에서 첫 번째 U+0027 (') 뒤에 오고 마지막 U+0027 (') 앞에 오는 문자 시퀀스입니다.

[expr.literal.byte-char.represented]

리터럴 표현식의 _표현된 문자(represented character)_는 다음과 같이 리터럴 내용으로부터 유도됩니다:

[expr.literal.byte-char.escape]
  • 리터럴 내용이 다음 형태의 이스케이프 시퀀스 중 하나인 경우, 표현된 문자는 해당 이스케이프 시퀀스의 이스케이프된 값입니다.
    • 단순 이스케이프
    • 8비트 이스케이프
[expr.literal.byte-char.single]
  • 그렇지 않은 경우, 표현된 문자는 리터럴 내용을 구성하는 단일 문자입니다.
[expr.literal.byte-char.result]

표현식의 값은 표현된 문자의 유니코드 스칼라 값입니다.

참고: 허용되는 바이트리터럴(BYTE_LITERAL) 토큰의 형식은 이러한 규칙들이 항상 u8 범위의 유니코드 스칼라 값을 가지는 단일 문자를 생성함을 보장합니다.

바이트 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
b'R';                              // 82
b'\'';                             // 39
b'\x52';                           // 82
b'\xA0';                           // 160
}
[expr.literal.byte-string]

바이트 문자열 리터럴 표현식

[expr.literal.byte-string.intro]

바이트 문자열 리터럴 표현식은 단일 바이트문자열리터럴(BYTE_STRING_LITERAL) 또는 원시바이트문자열리터럴(RAW_BYTE_STRING_LITERAL) 토큰으로 구성됩니다.

[expr.literal.byte-string.type]

표현식의 타입은 요소 타입이 u8인 배열에 대한 공유 참조(static 라이프타임을 가짐)입니다. 즉, 타입은 &'static [u8; N]이며, 여기서 N은 아래에서 설명하는 표현된 문자열의 바이트 수입니다.

[expr.literal.byte-string.no-suffix]

토큰은 접미사(suffix)를 가져서는 안 됩니다.

[expr.literal.byte-string.literal-content]

토큰의 _리터럴 내용(literal content)_은 토큰의 문자열 표현에서 첫 번째 U+0022 (") 뒤에 오고 마지막 U+0022 (") 앞에 오는 문자 시퀀스입니다.

[expr.literal.byte-string.represented]

리터럴 표현식의 _표현된 문자열(represented string)_은 다음과 같이 리터럴 내용으로부터 유도된 문자 시퀀스입니다:

[expr.literal.byte-string.escape]

  • 토큰이 바이트문자열리터럴(BYTE_STRING_LITERAL)인 경우, 리터럴 내용에서 나타나는 다음 형태의 각 이스케이프 시퀀스는 해당 이스케이프 시퀀스의 이스케이프된 값으로 대체됩니다.

    • 단순 이스케이프
    • 8비트 이스케이프
    • 문자열 연속 이스케이프 이러한 대체는 왼쪽에서 오른쪽 순서로 발생합니다. 예를 들어, b"\\x41" 토큰은 문자 \, x, 4, 1로 변환됩니다.
[expr.literal.byte-string.raw]
  • 토큰이 원시바이트문자열리터럴(RAW_BYTE_STRING_LITERAL)인 경우, 표현된 문자열은 리터럴 내용과 동일합니다.
[expr.literal.byte-string.result]

표현식의 값은 표현된 문자열 속 문자들의 유니코드 스칼라 값들을 동일한 순서로 포함하는 정적으로 할당된 배열에 대한 참조입니다.

참고: 허용되는 바이트문자열리터럴(BYTE_STRING_LITERAL) 및 원시바이트문자열리터럴(RAW_BYTE_STRING_LITERAL) 토큰의 형식은 이러한 규칙들이 항상 u8 범위의 배열 요소 값을 생성함을 보장합니다.

바이트 문자열 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
b"foo"; br"foo";                     // foo
b"\"foo\""; br#""foo""#;             // "foo"

b"foo #\"# bar";
br##"foo #"# bar"##;                 // foo #"# bar

b"\x52"; b"R"; br"R";                // R
b"\\x52"; br"\x52";                  // \x52
}
[expr.literal.c-string]

C 문자열 리터럴 표현식

[expr.literal.c-string.intro]

C 문자열 리터럴 표현식은 단일 C문자열리터럴(C_STRING_LITERAL) 또는 원시C문자열리터럴(RAW_C_STRING_LITERAL) 토큰으로 구성됩니다.

[expr.literal.c-string.type]

표현식의 타입은 표준 라이브러리 CStr 타입에 대한 공유 참조(static 라이프타임을 가짐)입니다. 즉, 타입은 &'static core::ffi::CStr입니다.

[expr.literal.c-string.no-suffix]

토큰은 접미사(suffix)를 가져서는 안 됩니다.

[expr.literal.c-string.literal-content]

토큰의 _리터럴 내용(literal content)_은 토큰의 문자열 표현에서 첫 번째 U+0022 (") 뒤에 오고 마지막 U+0022 (") 앞에 오는 문자 시퀀스입니다.

[expr.literal.c-string.represented]

리터럴 표현식의 _표현된 바이트(represented bytes)_는 다음과 같이 리터럴 내용으로부터 유도된 바이트 시퀀스입니다:

[expr.literal.c-string.escape]
  • 토큰이 C문자열리터럴(C_STRING_LITERAL)인 경우, 리터럴 내용은 \가 아닌 단일 유니코드 문자 또는 이스케이프인 항목들의 시퀀스로 취급됩니다. 항목들의 시퀀스는 다음과 같이 바이트 시퀀스로 변환됩니다:
    • 각 단일 유니코드 문자는 자신의 UTF-8 표현을 기여합니다.
    • 각 단순 이스케이프는 이스케이프된 값의 유니코드 스칼라 값을 기여합니다.
    • 각 8비트 이스케이프는 이스케이프된 값의 유니코드 스칼라 값을 포함하는 단일 바이트를 기여합니다.
    • 각 유니코드 이스케이프는 이스케이프된 값의 UTF-8 표현을 기여합니다.
    • 각 문자열 연속 이스케이프는 어떠한 바이트도 기여하지 않습니다.
[expr.literal.c-string.raw]
  • 토큰이 원시C문자열리터럴(RAW_C_STRING_LITERAL)인 경우, 표현된 바이트는 리터럴 내용의 UTF-8 인코딩입니다.

참고: 허용되는 C문자열리터럴(C_STRING_LITERAL) 및 원시C문자열리터럴(RAW_C_STRING_LITERAL) 토큰의 형식은 표현된 바이트에 널 바이트가 포함되지 않음을 보장합니다.

[expr.literal.c-string.result]

표현식의 값은 표현된 바이트 뒤에 널 바이트가 뒤따르는 바이트 배열을 포함하는 정적으로 할당된 CStr에 대한 참조입니다.

C 문자열 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
c"foo"; cr"foo";                     // foo
c"\"foo\""; cr#""foo""#;             // "foo"

c"foo #\"# bar";
cr##"foo #"# bar"##;                 // foo #"# bar

c"\x52"; c"R"; cr"R";                // R
c"\\x52"; cr"\x52";                  // \x52

c"æ";                                // 라틴어 소문자 AE (U+00E6)
c"\u{00E6}";                         // 라틴어 소문자 AE (U+00E6)
c"\xC3\xA6";                         // 라틴어 소문자 AE (U+00E6)

c"\xE6".to_bytes();                  // [230]
c"\u{00E6}".to_bytes();              // [195, 166]
}
[expr.literal.int]

정수 리터럴 표현식

[expr.literal.int.intro]

정수 리터럴 표현식은 단일 정수리터럴(INTEGER_LITERAL) 토큰으로 구성됩니다.

[expr.literal.int.suffix]

토큰에 접미사(suffix)가 있는 경우, 접미사는 기본 정수 타입 중 하나인 u8, i8, u16, i16, u32, i32, u64, i64, u128, i128, usize, 또는 isize여야 하며, 표현식은 해당 타입을 가집니다.

[expr.literal.int.infer]

토큰에 접미사가 없는 경우, 표현식의 타입은 타입 추론에 의해 결정됩니다.

[expr.literal.int.inference-unique-type]
  • 주변 프로그램 문맥으로부터 정수 타입을 유일하게 결정할 수 있다면, 표현식은 해당 타입을 가집니다.
[expr.literal.int.inference-default]
  • 프로그램 문맥이 타입을 충분히 제약하지 않는 경우, 기본값으로 부호 있는 32비트 정수인 i32가 사용됩니다.
[expr.literal.int.inference-error]
  • 프로그램 문맥이 타입을 과도하게 제약하는 경우, 정적 타입 에러로 간주됩니다.

정수 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
123;                               // i32 타입
123i32;                            // i32 타입
123u32;                            // u32 타입
123_u32;                           // u32 타입
let a: u64 = 123;                  // u64 타입

0xff;                              // i32 타입
0xff_u8;                           // u8 타입

0o70;                              // i32 타입
0o70_i16;                          // i16 타입

0b1111_1111_1001_0000;             // i32 타입
0b1111_1111_1001_0000i64;          // i64 타입

0usize;                            // usize 타입
}
[expr.literal.int.representation]

표현식의 값은 다음과 같이 토큰의 문자열 표현으로부터 결정됩니다:

[expr.literal.int.radix]

  • 정수 기수(radix)는 문자열의 처음 두 문자를 조사하여 다음과 같이 결정됩니다:

    • 0b는 기수 2를 나타냅니다.
    • 0o는 기수 8을 나타냅니다.
    • 0x는 기수 16을 나타냅니다.
    • 그 외의 경우 기수는 10입니다.
[expr.literal.int.radix-prefix-stripped]
  • 기수가 10이 아닌 경우, 처음 두 문자가 문자열에서 제거됩니다.
[expr.literal.int.type-suffix-stripped]
  • 모든 접미사는 문자열에서 제거됩니다.
[expr.literal.int.separators-stripped]
  • 모든 밑줄(_)은 문자열에서 제거됩니다.
[expr.literal.int.u128-value]
  • 문자열은 선택된 기수와 함께 u128::from_str_radix를 사용한 것처럼 u128 값으로 변환됩니다. 만약 값이 u128에 맞지 않으면 컴파일 에러입니다.
[expr.literal.int.cast]
  • u128 값은 숫자 캐스트(numeric cast)를 통해 표현식의 타입으로 변환됩니다.

참고: 최종 캐스트는 리터럴의 값이 표현식의 타입에 맞지 않을 경우 값을 잘라냅니다(truncate). rustc에는 overflowing_literals라는 이름의 린트 체크가 포함되어 있으며 기본값은 deny로, 이러한 상황이 발생하는 표현식을 거부합니다.

참고: 예를 들어 -1i8은 리터럴 표현식 1i8에 부정 연산자(negation operator)를 적용한 것이지, 단일 정수 리터럴 표현식이 아닙니다. 부호 있는 타입에서 가장 작은 음수 값을 표현하는 방식에 대한 노트는 오버플로(Overflow)를 참조하세요.

[expr.literal.float]

부동 소수점 리터럴 표현식

[expr.literal.float.intro]

부동 소수점 리터럴 표현식은 다음 두 가지 형식 중 하나를 가집니다:

  • 단일 부동소수점리터럴(FLOAT_LITERAL) 토큰
  • 접미사가 있고 기수 표시가 없는 단일 정수리터럴(INTEGER_LITERAL) 토큰
[expr.literal.float.suffix]

토큰에 접미사(suffix)가 있는 경우, 접미사는 기본 부동 소수점 타입 중 하나인 f32 또는 f64여야 하며, 표현식은 해당 타입을 가집니다.

[expr.literal.float.infer]

토큰에 접미사가 없는 경우, 표현식의 타입은 타입 추론에 의해 결정됩니다.

[expr.literal.float.inference-unique-type]
  • 주변 프로그램 문맥으로부터 부동 소수점 타입을 유일하게 결정할 수 있다면, 표현식은 해당 타입을 가집니다.
[expr.literal.float.inference-default]
  • 프로그램 문맥이 타입을 충분히 제약하지 않는 경우, 기본값으로 f64가 사용됩니다.
[expr.literal.float.inference-error]
  • 프로그램 문맥이 타입을 과도하게 제약하는 경우, 정적 타입 에러로 간주됩니다.

부동 소수점 리터럴 표현식의 예시:

#![allow(unused)]
fn main() {
123.0f64;        // f64 타입
0.1f64;          // f64 타입
0.1f32;          // f32 타입
12E+99_f64;      // f64 타입
5f32;            // f32 타입
let x: f64 = 2.; // f64 타입
}
[expr.literal.float.result]

표현식의 값은 다음과 같이 토큰의 문자열 표현으로부터 결정됩니다:

[expr.literal.float.type-suffix-stripped]
  • 모든 접미사는 문자열에서 제거됩니다.
[expr.literal.float.separators-stripped]
  • 모든 밑줄(_)은 문자열에서 제거됩니다.
[expr.literal.float.value]
  • 문자열은 f32::from_str 또는 f64::from_str을 사용한 것처럼 표현식의 타입으로 변환됩니다.

참고: 예를 들어 -1.0은 리터럴 표현식 1.0에 부정 연산자(negation operator)를 적용한 것이지, 단일 부동 소수점 리터럴 표현식이 아닙니다.

참고: inf와 NaN은 리터럴 토큰이 아닙니다. 리터럴 표현식 대신 f32::INFINITY, f64::INFINITY, f32::NAN, f64::NAN 상수들을 사용할 수 있습니다. rustc에서 무한대로 평가될 만큼 큰 리터럴은 overflowing_literals 린트 체크를 발생시킵니다.

[expr.literal.bool]

불리언 리터럴 표현식

[expr.literal.bool.intro]

불리언 리터럴 표현식은 true 또는 false 키워드 중 하나로 구성됩니다.

[expr.literal.bool.result]

표현식의 타입은 기본 불리언 타입이며, 그 값은 다음과 같습니다:

  • 키워드가 true인 경우 true
  • 키워드가 false인 경우 false
