The Rust Edition Guide

프렐루드에 추가된 항목

요약

  • TryInto, TryFrom 및 FromIterator 트레이트는 이제 프렐루드의 일부입니다.
  • 이로 인해 트레이트 메서드 호출이 모호해져 일부 코드가 컴파일되지 않을 수 있습니다.

세부 정보

표준 라이브러리의 프렐루드는 모든 모듈에서 자동으로 가져오는 모든 것을 포함하는 모듈입니다. Option, Vec, drop 및 Clone과 같이 일반적으로 사용되는 항목을 포함합니다.

Rust 컴파일러는 프렐루드에 추가된 항목이 기존 코드를 깨뜨리지 않도록 하기 위해 프렐루드의 항목보다 수동으로 가져온 항목을 우선시합니다. 예를 들어, pub struct Option;을 포함하는 example이라는 크레이트나 모듈이 있는 경우 use example::*;은 Option이 표준 라이브러리의 것이 아니라 example의 것을 명확하게 참조하도록 합니다.

그러나 프렐루드에 _트레이트_를 추가하면 미묘한 방식으로 기존 코드가 깨질 수 있습니다. 예를 들어, MyTryInto 트레이트에서 오는 x.try_into() 호출은 std의 TryInto도 가져온 경우 컴파일에 실패할 수 있습니다. try_into 호출이 이제 모호하고 어느 트레이트에서든 올 수 있기 때문입니다. 이것이 우리가 아직 TryInto를 프렐루드에 추가하지 않은 이유입니다. 이런 식으로 깨질 코드가 많기 때문입니다.

해결책으로 Rust 2021은 새로운 프렐루드를 사용합니다. 세 가지 새로운 추가 사항을 제외하고는 현재 것과 동일합니다.

  • std::convert::TryInto
  • std::convert::TryFrom
  • std::iter::FromIterator

추적 문제는 여기에서 찾을 수 있습니다.

마이그레이션

2021 에디션의 일부로 Rust 2018 코드베이스를 Rust 2021로 자동 마이그레이션하는 데 도움이 되도록 마이그레이션 린트인 rust_2021_prelude_collisions가 추가되었습니다.

코드를 Rust 2021 에디션과 호환되도록 마이그레이션하려면 다음을 실행하십시오.

cargo fix --edition

린트는 새 프렐루드 트레이트 중 하나에 정의된 메서드와 이름이 같은 함수나 메서드가 호출되는 경우를 감지합니다. 경우에 따라 이전에 호출했던 것과 동일한 함수를 계속 호출하도록 호출을 다양한 방식으로 다시 작성할 수 있습니다.

코드를 수동으로 마이그레이션하거나 cargo fix가 수행하는 작업을 더 잘 이해하고 싶다면 아래에 마이그레이션이 필요한 상황과 필요하지 않은 경우의 반대 예를 간략하게 설명했습니다.

마이그레이션 필요

충돌하는 트레이트 메서드

범위에 있는 두 트레이트의 메서드 이름이 같으면 어떤 트레이트 메서드를 사용해야 할지 모호합니다. 예를 들어:

trait MyTrait<A> {
  // 이 이름은 `std`의 `FromIterator` 트레이트에 있는 `from_iter` 메서드와 같습니다. 
  fn from_iter(x: Option<A>);
}

impl<T> MyTrait<()> for Vec<T> {
  fn from_iter(_: Option<()>) {}
}

fn main() {
  // Vec<T>는 `std::iter::FromIterator`와 `MyTrait`를 모두 구현합니다
  // 두 트레이트가 모두 범위에 있는 경우(Rust 2021의 경우와 같음),
  // 그러면 어떤 `from_iter` 메서드를 호출해야 할지 모호해집니다
  <Vec<i32>>::from_iter(None);
}

완전 정규화된 구문을 사용하여 이 문제를 해결할 수 있습니다.

fn main() {
  // 이제 어떤 트레이트 메서드를 참조하는지 명확합니다
  <Vec<i32> as MyTrait<()>>::from_iter(None);
}

dyn Trait 객체의 고유 메서드

일부 사용자는 dyn Trait 값에 대해 메서드를 호출하는데, 이 메서드 이름이 새로운 prelude 특성과 겹칩니다.

#![allow(unused)]
fn main() {
mod submodule {
  pub trait MyTrait {
    // 이것은 `TryInto::try_into`와 같은 이름을 갖습니다.
    fn try_into(&self) -> Result<u32, ()>;
  }
}

// `MyTrait`는 여기 범위에 포함되지 않으며 경로 `submodule::MyTrait`
"을 통해서만 참조될 수 있습니다fn bar(f: Box<dyn submodule::MyTrait>) {
  // `std::convert::TryInto`가 범위 내에 있는 경우(Rust의 경우처럼)
2021",
 // 어떤 `try_into` 메서드를 호출해야 할지 모호해집니다
  f.try_into();
}
}

Unlike with static dispatch methods, calling a trait method on a trait object does not require that the trait be in scope. The code above works as long as there is no trait in scope with a conflicting method name. When the TryInto trait is in scope (which is the case in Rust 2021), this causes an ambiguity. Should the call be to MyTrait::try_into or std::convert::TryInto::try_into?

이런 경우에는 추가적인 역참조를 추가하거나, 메서드 수신자의 타입을 명확히 하여 이 문제를 해결할 수 있습니다. 이렇게 하면 prelude 특성의 메서드 대신 dyn Trait 메서드가 선택됩니다. 예를 들어, 위의 "f.try_into()를 "(&*f).try_into()로 변환하면 dyn MyTrait에서 MyTrait::try_into 메서드만 참조할 수 있는 try_into를 호출하게 됩니다.

마이그레이션이 필요 없습니다

고유한 방법

많은 타입이 특성과 동일한 이름을 가진 자체 내장 메서드를 정의합니다.메서드입니다. 예를 들어, 아래 구조체 MyStruct는 from_iter를 구현합니다.이 메서드는 표준 라이브러리에 있는 특성 FromIterator의 메서드와 동일한 이름을 공유합니다.

#![allow(unused)]
fn main() {
use std::iter::IntoIterator;

struct MyStruct {
  data: Vec<u32>
}

impl MyStruct {
  // 이것은 `std::iter::FromIterator::from_iter`와 이름이 같습니다
  fn from_iter(iter: impl IntoIterator<Item = u32>) -> Self {
    Self {
      data: iter.into_iter().collect()
    }
  }
}

impl std::iter::FromIterator<u32> for MyStruct {
    fn from_iter<I: IntoIterator<Item = u32>>(iter: I) -> Self {
      Self {
        data: iter.into_iter().collect()
      }
    }
}
}

고유 메서드는 항상 트레이트 메서드보다 우선하므로 마이그레이션이 필요하지 않습니다.

구현 참조

린트는 2021 에디션을 코드베이스에 도입하는 것이 이름 확인 충돌을 일으킬지(따라서 에디션 변경 후 코드가 깨지는지) 여부를 결정할 때 몇 가지 요소를 고려해야 합니다. 이러한 요소는 다음과 같습니다.

  • 호출이 완전 정규화된 호출입니까, 아니면 점-호출 메서드 구문을 사용합니까?
    • 이것은 메서드 호출 구문에서 자동 참조 및 자동 역참조로 인해 이름이 확인되는 방식에 영향을 줍니다. 수동으로 역참조/참조하면 점-호출 메서드 구문의 경우 우선순위를 지정할 수 있지만, 완전 정규화된 호출은 메서드 경로에서 유형과 트레이트 이름을 지정해야 합니다(예: <Type as Trait>::method).
  • 이것은 고유 메서드입니까, 아니면 트레이트 메서드입니까?
    • self를 사용하는 고유 메서드는 TryInto::try_into보다 우선합니다. 고유 메서드가 트레이트 메서드보다 우선하기 때문입니다. 그러나 &self 또는 &mut self를 사용하는 고유 메서드는 자동 참조가 필요하기 때문에 우선하지 않습니다(TryInto::try_into는 self를 사용하므로 그렇지 않습니다).
  • 이 메서드의 출처는 core/std입니까? (트레이트는 자신과 충돌할 수 없으므로)
  • 주어진 유형이 충돌할 수 있는 트레이트를 구현합니까?
  • 메서드가 동적 디스패치를 통해 호출됩니까? (즉, self 유형이 dyn Trait입니까)
    • 그렇다면 트레이트 가져오기는 확인에 영향을 미치지 않으며 마이그레이션 린트가 발생할 필요가 없습니다.
